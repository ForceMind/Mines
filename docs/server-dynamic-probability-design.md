# 服务器动态概率调整方案

## 1. 文档目的

本文档用于说明 Mines 游戏中“服务器动态调整概率”的产品设计方案，方便产品、后端、前端、运营和测试人员对齐实现目标与行为边界。

本文只描述设计和方法，不包含代码实现。

## 2. 设计目标

动态概率调整的核心目标是：在不让用户累计 RTP 超过配置上限的前提下，对短期体验过差的用户进行一定程度的保护。

具体目标如下：

1. 游戏结果由服务器统一判定，前端不参与概率计算和结果生成。
2. 每一次玩家点击格子，都由服务器实时判断本次是否安全。
3. 当用户 RTP 低于保护阈值时，服务器可以在本次原本会失败的情况下，按保护规则将本次结果调整为安全。
4. 任何保护都不能让用户累计 RTP 超过后台配置的 RTP 上限，默认上限为 100%。
5. 所有保护行为必须在后台可查看、可统计、可追踪。

## 3. 核心概念

### 用户 RTP

用户 RTP 表示某个用户累计获得的派彩金额与累计下注金额之间的比例。

它用于判断用户当前是否处于明显低于预期的状态。

示例：

| 累计下注 | 累计派彩 | 用户 RTP |
|---:|---:|---:|
| 1000 金币 | 700 金币 | 70% |
| 1000 金币 | 850 金币 | 85% |
| 1000 金币 | 1000 金币 | 100% |

### 保护阈值

保护阈值是触发动态概率保护的下限。

默认建议为 80%。

当用户 RTP 低于该值时，服务器认为该用户短期体验较差，可以进入保护判断。

### RTP 上限

RTP 上限用于限制保护的边界。

默认建议为 100%。

无论是否触发保护，用户累计派彩都不能超过累计下注所允许的上限。

### 保护偏离值

保护偏离值表示“当本次原始随机结果为失败时，服务器把本次结果改为安全的概率”。

该值由后台配置，可以根据用户 RTP 偏低程度进行动态放大，但不能超过后台配置的最大值。

## 4. 游戏结果判定方式

Mines 不在开局时固定生成完整雷图。

每次玩家点击格子时，服务器根据当前局面实时判断本次结果。

服务器需要知道以下信息：

1. 棋盘总格子数。
2. 本局雷数。
3. 当前已经安全打开的格子数。
4. 当前剩余未打开格子数。
5. 当前剩余安全格子数。
6. 当前用户累计下注、累计派彩和 RTP。

在每次点击时，服务器先按正常概率做一次基础随机。

如果基础随机结果为安全，则直接返回安全。

如果基础随机结果为失败，则进入动态保护判断。

## 5. 动态保护判断流程

动态保护只在“本次基础随机结果为失败”时发生。

完整判断流程如下：

1. 玩家点击一个未打开格子。
2. 服务器计算当前正常概率。
3. 服务器进行一次基础随机，得到本次原始结果。
4. 如果原始结果为安全，直接返回安全。
5. 如果原始结果为失败，服务器检查用户 RTP 是否低于保护阈值。
6. 如果用户 RTP 不低于保护阈值，返回失败。
7. 如果用户 RTP 低于保护阈值，服务器继续检查本次如果改为安全，当前可提现金额是否会导致用户累计 RTP 超过上限。
8. 如果会超过 RTP 上限，返回失败。
9. 如果不会超过 RTP 上限，服务器按保护偏离值再做一次保护判断。
10. 如果保护判断命中，本次结果改为安全。
11. 如果保护判断未命中，本次仍返回失败。

该流程保证保护只影响“原本失败的点击”，不会改变正常已经安全的结果，也不会突破用户 RTP 上限。

## 6. 保护触发条件

保护必须同时满足以下条件：

1. 本次基础随机结果为失败。
2. 用户当前 RTP 低于后台配置的保护阈值。
3. 当前局仍在进行中。
4. 用户本局保护次数未超过后台限制。
5. 本次如果改为安全，不会导致用户累计 RTP 超过上限。
6. 本次保护偏离判断命中。

只要其中任一条件不满足，本次不触发保护。

## 7. RTP 上限控制方式

RTP 上限是整个方案的硬边界。

服务器在两个关键节点做控制：

1. 点击保护前判断：如果本次改为安全后，对应的当前可提现金额会让用户超过 RTP 上限，则不允许保护。
2. 派彩结算时判断：如果用户提现或全清后理论派彩超过 RTP 上限允许范围，则按上限进行封顶。

这两个控制点可以避免以下情况：

1. 用户因保护机制获得超过累计下注的长期收益。
2. 用户在低 RTP 状态下被连续保护后直接突破 100% RTP。
3. 配置错误导致超额派彩。

## 8. 后台配置项

后台需要支持以下配置：

| 配置项 | 说明 | 建议默认值 |
|---|---|---:|
| 保护触发 RTP 下限 | 用户 RTP 低于该值才允许进入保护判断 | 80% |
| 用户 RTP 上限 | 用户累计 RTP 不允许超过该值 | 100% |
| 基础保护偏离值 | 低 RTP 用户的基础保护概率 | 12% |
| 最大保护偏离值 | 单次保护概率的最大值 | 65% |
| 单局最大保护次数 | 一局中最多允许保护多少次 | 3 |
| 下注档位 | 玩家可选择的下注金额档位 | 1, 5, 10, 20, 50, 100 |
| 平台优势 | 正常派彩计算中的平台优势参数 | 3% |

后台配置修改后，应对新局或后续点击生效，具体生效时机由开发实现时统一。

## 9. 保护偏离值的动态放大

保护偏离值可以根据用户 RTP 低于保护阈值的程度进行放大。

设计原则如下：

1. 用户 RTP 越低，保护偏离值越高。
2. 偏离值不能超过后台配置的最大保护偏离值。
3. 偏离值只影响“原本失败”的点击。
4. 偏离值不是直接提高所有点击的安全概率，而是在失败结果出现后给一次保护机会。
5. 偏离值应有单局次数限制，避免一局内连续保护过多。

示例：

| 用户 RTP 状态 | 保护倾向 |
|---|---|
| 高于 80% | 不保护 |
| 70% 到 80% | 轻度保护 |
| 50% 到 70% | 中度保护 |
| 低于 50% | 较高保护，但不能超过最大保护偏离值 |

## 10. 前端展示原则

前端不展示 RTP、保护偏离值或后台保护状态。

玩家前端只展示正常游戏信息：

1. 棋盘。
2. 金币余额。
3. 下注档位。
4. 当前可提现金额。
5. 游戏结果弹窗。

前端不参与概率计算，不保存雷图，不决定输赢。

所有点击结果都必须以服务器响应为准。

## 11. 后台观测和日志

后台需要展示以下数据，方便运营和开发排查：

1. 玩家 ID。
2. 玩家余额。
3. 累计下注。
4. 累计派彩。
5. 当前 RTP。
6. 游戏局号。
7. 每局下注金额。
8. 每局状态。
9. 每局安全点击次数。
10. 当前局是否启用保护。
11. 本局保护触发次数。
12. 每次保护发生的点击位置、原始随机结果、保护偏离值和最终结果。

后台日志用于审计和问题定位，不展示给普通玩家。

## 12. 风控边界

该方案必须遵守以下边界：

1. 不允许用户累计 RTP 超过后台配置上限。
2. 不允许前端传入自定义概率。
3. 不允许前端传入自定义派彩。
4. 不允许前端决定本次是否安全。
5. 不允许绕过服务器直接结算。
6. 后台配置修改需要记录操作时间和配置值。
7. 建议保留保护触发日志，便于后续核对。

## 13. 测试重点

测试需要覆盖以下场景：

1. 用户 RTP 高于保护阈值时，不触发保护。
2. 用户 RTP 低于保护阈值时，失败结果可能被保护为安全。
3. 保护后如果会超过 RTP 上限，则不允许保护。
4. 提现时派彩不能让用户 RTP 超过上限。
5. 全清时派彩不能让用户 RTP 超过上限。
6. 单局保护次数不能超过后台配置。
7. 后台修改保护配置后，新配置按预期生效。
8. 下注金额只能使用后台配置的档位。
9. 刷新页面或服务重启后，用户数据和局记录仍可在后台查看。

## 14. 方案总结

服务器动态概率调整不是提前移动雷，也不是前端决定输赢。

该方案采用“每次点击服务器实时随机”的方式，并在原始结果为失败时，根据用户 RTP、保护阈值、保护偏离值、单局次数限制和 RTP 上限进行二次判断。

最终目标是在用户短期体验过差时提供有限保护，同时确保整体派彩不会突破后台配置的 RTP 上限。
