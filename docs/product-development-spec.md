# Mines 产品开发说明文档

## 1. 文档说明

本文档是 Mines 项目的完整产品开发说明，面向产品、前端、后端、测试、运维和新加入项目的开发人员。

文档目标是让不了解项目的人，也能读懂这个产品要做什么、页面怎么工作、服务器怎么判断结果、后台怎么配置、数据怎么保存、部署脚本怎么使用，以及测试时需要重点检查什么。

本文档只使用自然语言描述，不写代码，不写伪代码。

## 2. 产品一句话说明

Mines 是一个 5x5 棋盘的概率游戏。

玩家使用金币下注，点击棋盘格子寻找钻石。如果点击到安全格，会显示钻石，并可以选择继续点击或提现。如果点击到地雷，本局失败。如果玩家在失败前选择提现，则获得当前可提现金币。

服务器负责每一次点击的结果判断，前端只负责展示和发送玩家操作。

## 3. 当前版本范围

当前版本包含以下内容：

1. 玩家游戏页面。
2. 后台管理页面。
3. 服务器游戏接口。
4. 服务器动态概率保护逻辑。
5. 用户金币、下注、派彩和 RTP 统计。
6. 运行数据本地持久化。
7. Linux 一键部署脚本。
8. 产品和开发说明文档。

当前版本暂不包含以下内容：

1. 用户注册登录。
2. 真实支付和充值。
3. 多语言切换。
4. 后台账号权限。
5. 远程数据库。
6. 正式风控审计系统。

## 4. 重要概念解释

### 4.1 金币

金币是玩家页面显示的游戏余额单位。

玩家下注使用金币，获得派彩也是金币。

前端不再显示美元符号，后台也使用“金币”作为金额单位。

### 4.2 棋盘

当前玩家页面固定使用 5x5 棋盘。

也就是说，一局游戏共有 25 个格子。

之前的棋盘大小切换功能已经临时取消，但相关位置保留了注释，方便以后恢复。

### 4.3 地雷数量

地雷数量表示本局中失败格子的数量。

玩家页面可以使用减号和加号调整地雷数量。

当前固定 5x5 棋盘，所以地雷数量最小为 1，最大为 24。

地雷越多，玩家每次点击安全格的概率越低，但成功后可提现金额也会更高。

### 4.4 下注档位

下注不允许玩家随便输入。

后台配置固定的下注档位，玩家页面只能使用减号和加号在这些档位之间切换。

例如后台配置为 1、5、10、20、50、100，那么玩家只能选择这些金额下注。

### 4.5 安全格

安全格是玩家点击后不会失败的格子。

玩家点击安全格后，前端显示亮色钻石。

### 4.6 地雷

地雷是玩家点击后会失败的格子。

玩家点击到地雷后，本局结束，前端弹窗提示失败，并显示炸弹。

### 4.7 Cash Out

Cash Out 表示玩家在本局还没有失败之前主动提现。

玩家至少打开一个安全格后，才允许 Cash Out。

### 4.8 RTP

RTP 可以理解为“玩家拿回了多少”。

在本项目中，用户 RTP 由累计派彩和累计下注计算得出。

如果玩家累计下注 1000 金币，累计拿回 800 金币，那么玩家 RTP 为 80%。

RTP 用于判断玩家是否需要保护，也用于限制玩家累计派彩不能超过后台设置的上限。

## 5. 产品角色

### 5.1 普通玩家

普通玩家只看到游戏页面。

玩家可以做以下操作：

1. 查看金币余额。
2. 调整地雷数量。
3. 调整下注档位。
4. 开始游戏。
5. 点击格子。
6. Cash Out。
7. 查看失败或获胜弹窗。

普通玩家看不到后台配置、RTP、动态概率保护记录和系统日志。

### 5.2 后台管理员

后台管理员通过手动输入后台地址访问后台页面。

当前版本暂时不需要令牌。

后台管理员可以查看：

1. 系统总余额。
2. 总下注。
3. 总派彩。
4. 系统 RTP。
5. 玩家列表。
6. 最近对局。
7. 保护触发情况。

后台管理员可以配置：

1. 平台优势。
2. 保护触发 RTP 下限。
3. 用户 RTP 上限。
4. 基础保护偏离值。
5. 最大保护偏离值。
6. 单局最大保护次数。
7. 下注档位。

## 6. 玩家页面说明

### 6.1 页面目标

玩家页面的目标是让玩家快速理解并进行游戏。

页面应该重点突出棋盘、金币余额、下注档位、地雷数量和主操作按钮。

页面不显示后台信息，不显示 RTP，不显示保护状态，不显示管理入口。

### 6.2 页面布局

玩家页面采用正方形主画面。

页面顶部显示：

1. 游戏名称 Mines。
2. 当前安全点击次数。
3. 下一次安全概率。
4. 当前可提现金币。
5. 玩家金币余额。

页面中间显示：

1. 固定 5x5 棋盘。
2. 未点击格子使用蓝色风格。
3. 安全格显示亮色钻石。
4. 地雷格显示炸弹。

页面底部显示：

1. 地雷数量减号。
2. 地雷数量当前值。
3. 地雷数量加号。
4. 下注档位减号。
5. 当前下注金币。
6. 下注档位加号。
7. 主操作按钮。

### 6.3 棋盘大小

当前版本固定为 5x5。

前端页面不展示棋盘大小选择。

原本的棋盘大小选择区域保留为注释，未来如果要恢复，可以重新打开该区域并恢复对应逻辑。

### 6.4 地雷数量交互

地雷数量使用减号和加号调整。

玩家不能手动输入任意值。

调整规则如下：

1. 默认地雷数量为 5。
2. 最小地雷数量为 1。
3. 最大地雷数量为 24。
4. 游戏进行中不能调整地雷数量。
5. 游戏结束后可以继续调整。

### 6.5 下注档位交互

下注使用后台配置的档位。

玩家页面只显示当前档位，使用减号和加号切换。

调整规则如下：

1. 默认档位由后台配置决定。
2. 当前默认档位为 10 金币。
3. 玩家不能手动输入任意下注金额。
4. 游戏进行中不能调整下注。
5. 后台修改档位后，玩家刷新页面或重新获取配置后生效。

### 6.6 主按钮状态

玩家页面只有一个主操作按钮。

按钮根据游戏状态变化：

1. 未开始时显示 Start Round。
2. 开局后显示 Cash Out。
3. 玩家尚未打开安全格时，Cash Out 不可用。
4. 玩家打开至少一个安全格后，Cash Out 可用。
5. 本局结束后显示 Next Round。
6. 玩家点击 Next Round 后开始下一局。

页面不显示 End Round。

### 6.7 结果弹窗

游戏结束时必须出现弹窗。

失败弹窗：

1. 显示炸弹图标。
2. 标题提示本局失败。
3. 说明玩家点击到了地雷。

获胜或提现弹窗：

1. 显示钻石图标。
2. 标题提示恭喜。
3. 显示获得多少金币。

弹窗关闭后，玩家可以点击 Next Round 开始下一局。

### 6.8 前端玩家 ID

玩家页面不显示 Player 输入框。

前端会在浏览器本地自动生成一个玩家 ID。

这个 ID 用于服务器区分不同玩家的余额、下注、派彩和 RTP。

普通玩家不需要知道这个 ID，也不需要手动输入。

## 7. 后台页面说明

### 7.1 后台访问方式

当前版本后台入口不显示在玩家页面。

后台管理员需要手动输入后台地址访问。

当前版本暂时取消令牌校验。

### 7.2 后台页面目标

后台页面用于配置系统参数、查看用户数据、查看对局记录和观察保护触发情况。

后台页面不面向普通玩家。

### 7.3 后台配置区域

后台配置区域包含以下配置项。

| 配置项 | 说明 |
|---|---|
| 平台优势 | 用于服务器内部概率计算的修正比例，不是最终金币扣减 |
| 保护触发 RTP 下限 | 用户 RTP 低于该值时，允许进入保护判断 |
| 用户 RTP 上限 | 用户累计 RTP 不能超过该值 |
| 基础保护偏离值 | 低 RTP 用户进入保护时的基础保护概率 |
| 最大保护偏离值 | 单次保护概率的上限 |
| 单局最大保护次数 | 一局中最多允许保护多少次 |
| 下注档位 | 玩家可选择的下注金额档位 |

### 7.4 后台统计区域

后台统计区域展示系统整体数据。

包括：

1. 总余额。
2. 总下注。
3. 总派彩。
4. 系统 RTP。

这些数据用于运营和开发观察整体状态。

### 7.5 玩家列表

玩家列表展示每个玩家的状态。

包括：

1. 玩家 ID。
2. 当前金币余额。
3. 累计下注。
4. 累计派彩。
5. 当前 RTP。
6. 已触发保护次数。

如果后台没有玩家数据，需要展示清晰提示，说明前台有玩家打开游戏后会自动创建记录。

### 7.6 最近对局

最近对局展示最近产生的游戏记录。

包括：

1. 局号。
2. 玩家 ID。
3. 对局状态。
4. 下注金币。
5. 安全点击次数。
6. 当前可提现金币。
7. 是否启用保护。
8. 保护触发次数。

如果暂时没有对局，需要展示清晰提示，说明玩家开始游戏后会显示最近对局。

## 8. 服务器职责说明

服务器是整个游戏的唯一结果判定方。

前端不能决定格子是否安全，也不能决定派彩金额。

服务器负责：

1. 创建玩家记录。
2. 保存玩家金币余额。
3. 接收开局请求。
4. 接收点击格子请求。
5. 判断本次点击结果。
6. 计算当前可提现金币。
7. 处理 Cash Out。
8. 判断是否触发 RTP 保护。
9. 限制用户 RTP 不能超过上限。
10. 保存局记录。
11. 向后台提供统计数据。

## 9. 游戏流程说明

### 9.1 玩家打开页面

玩家打开页面后，前端会检查浏览器本地是否已经有玩家 ID。

如果没有，前端会自动生成一个玩家 ID。

前端随后向服务器请求当前玩家状态。

服务器如果发现该玩家不存在，会创建新玩家并给默认金币余额。

### 9.2 玩家开始游戏

玩家点击 Start Round 后，前端把以下信息发送给服务器：

1. 玩家 ID。
2. 固定棋盘大小 5x5。
3. 当前地雷数量。
4. 当前下注档位。

服务器收到请求后检查：

1. 玩家是否存在。
2. 玩家是否已有进行中的局。
3. 下注是否属于后台配置档位。
4. 玩家余额是否足够。
5. 地雷数量是否在允许范围内。

检查通过后，服务器扣除下注金币，创建新局，并返回当前局状态。

### 9.3 玩家点击格子

玩家点击格子后，前端只告诉服务器点击了哪个位置。

服务器根据当前局面判断本次是否安全。

服务器返回结果后，前端更新棋盘。

如果安全，格子显示钻石。

如果失败，本局结束，前端显示炸弹和失败弹窗。

### 9.4 玩家 Cash Out

玩家至少打开一个安全格后，Cash Out 按钮可用。

玩家点击 Cash Out 后，服务器根据当前局面计算派彩。

可提现金币按连续安全总概率计算，不在最终金币上乘以平台优势。

派彩不能让玩家累计 RTP 超过后台配置上限。

服务器结算后，本局结束，前端弹窗显示获得多少金币。

### 9.5 玩家全清

如果玩家打开了所有安全格，本局自动结束。

服务器进行全清结算。

前端弹窗提示恭喜并展示获得金币。

## 10. 概率判断说明

当前版本不在开局时生成完整雷图。

每次玩家点击格子时，服务器根据当前局面实时判断本次结果。

服务器知道当前还剩多少未打开格子，也知道理论上还有多少安全格。

玩家已经打开的格子都视为安全格。

每次点击时，服务器先计算当前基础安全概率。

如果后台平台优势为 X，服务器内部用于基础判断的安全概率要再乘以 1 - X。

例如基础安全概率是 80%，平台优势是 3%，服务器内部安全概率就是 80% × 97%，也就是 77.6%。

内部失败概率等于 100% 减去内部安全概率。

平台优势不是最终金币扣减，不能在 Cash Out 或全清结算时再把获得金币乘以 1 - X。

如果本次基础判断是安全，就直接返回安全。

如果本次基础判断是失败，服务器再判断是否允许进入动态保护流程。

## 11. 动态 RTP 保护说明

动态 RTP 保护用于改善用户短期体验过差的情况。

该保护不是移动地雷，也不是前端决定输赢。

保护只发生在服务器本次基础判断原本为失败时。

保护需要同时满足以下条件：

1. 用户当前 RTP 低于后台配置的保护阈值。
2. 本局保护次数没有超过后台配置上限。
3. 如果本次改为安全，不会让用户累计 RTP 超过上限。
4. 本次保护偏离判断命中。

如果这些条件都满足，服务器可以把本次结果改为安全。

如果任意条件不满足，本次仍然失败。

## 12. RTP 上限说明

用户 RTP 上限是硬限制。

默认上限为 100%。

服务器需要在两个地方控制 RTP：

1. 判断是否允许保护时。
2. 实际派彩结算时。

如果某次保护或派彩会导致用户累计 RTP 超过上限，服务器不能放开。

如果理论派彩超过上限允许金额，服务器需要按上限封顶。

## 13. 数据保存说明

当前版本使用本地文件保存运行数据。

保存内容包括：

1. 后台配置。
2. 玩家数据。
3. 游戏局记录。
4. 点击决策记录。
5. 保护触发记录。

服务器重启后，会读取本地保存的数据。

这样后台不会因为服务重启而清空。

部署到 Linux 后，运行数据默认保存在应用目录下的数据目录。

## 20. 总结

当前 Mines 产品的核心设计是：前端负责展示和交互，后端负责结果、概率、派彩和数据。

玩家页面保持简单，只让玩家关注金币、地雷、下注、棋盘和结果。

后台页面负责配置和观察。

服务器动态概率保护用于改善低 RTP 用户体验，但必须严格受 RTP 上限控制。

整个项目当前适合作为可部署、可演示、可继续扩展的基础版本。
